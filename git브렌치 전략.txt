[git 브랜치를 운영하기 위한 전략]

1. gitflow
  -1) main(master)브랜치
    => 실제 제품으로 나갈 수 있는 상태
    
  -2) develop
   => 메인 브랜치를 기반으로 만들어진 브랜치
   => 다음 배포에 나갈 내요을 메인에서 머지할 수 없으므로 진행중인 개발내용이 머지될 브랜치
   => 다음 출시 버전을 개발하는 브랜치
   
  -3) feature
   => 각각의 개발자들이 자기의 개발을 하기 위해서 만드는 작업 브랜치
   => 여기서 작업이 끝나고 푸쉬하면 merge가 끝나면 삭제할 브랜치
                            =====
                            develop
    => 기능을 개발하는 브랜치
                            
  -4) release
   => 이번 출시 버전을 준비하는 브랜치로 devleop 브랜치를 기반으로 준비
   => 그다음에 작업이 완료된 후 배포시점이 왔을 때 생성
   => Q&A 나 여러 작업을 마무리하고 main 브랜치와 merge

  -5) hotfix
   => 배포직전에 혹은 운영중에 긴급하게 버그, 오류(이슈)가 발생하는 경우 아룰 정리하는 브랜치
   => 출시 직전에 버그를 수정하기 위한 브랜치
   => release브랜치를 기준으로 생성 
  => 주로 main 과 merge한다
  
    => 대략 기업들은 총 5개의 브랜치를 이용해 작업한다.
